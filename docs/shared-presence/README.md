# Shared Presence
## Collective Visibility for Group Projects

---

## What This Section Is

This section of the Vestigia repository supports groups working on final-year
projects who want to build a shared, outward-facing presence for their work.

A shared presence is any intentional, collective artefact that communicates
what your project is, who is behind it, and how it is progressing — to clients,
supervisors, and external readers including potential employers.

It is not a replacement for individual documentation or individual showcases.
It is the collective layer that connects individual work to a shared project context.

---

## What a Shared Presence Is Not

A shared presence is not:

◆ a submission requirement  
◇ an assessed deliverable  
◆ a marketing page created at the end of the project  
◇ a replacement for Guide A records or Guide B individual showcases

Participation is entirely optional. If your group chooses to build one,
it should serve you — not exist for the sake of existing.

---

## How This Section Relates to Guide A and Guide B

Guide A supports individual record-keeping during the project.  
Guide B supports individual showcases after the project.

A shared presence sits alongside both:

◆ it draws from the same project work without duplicating individual records  
◇ it provides a shared anchor that individual Guide B showcases can reference  
◆ it gives clients and supervisors visibility without requiring access to private records  
◇ it demonstrates collective professionalism to external readers

Each team member's individual showcase remains their own. The shared presence
provides the group context that makes individual contributions understandable.

## What This Section Contains

**01 — What Is a Shared Presence**  
The conceptual foundation. What a shared presence does, why it matters,
what distinguishes a meaningful one from an empty one, and how to think
about it across different project types.

Read this first, regardless of your discipline or platform choice.

**02 — Building a Project Website**  
Specific guidance for Computer Science, IT, and Engineering groups building
a live project website using GitHub Pages. Covers what to include, how to
publish, and what makes a group site genuinely useful rather than decorative.

**03 — Shared Presence for Other Disciplines**  
Guidance for groups in design, data science, education, business, and other
fields — including platform options appropriate to each discipline and the
content considerations specific to each type of project.

## Where to Start

If your group is in CS, IT, or Engineering and is comfortable building a website:
start with **01**, then move to **02**.

If your group is in another discipline or prefers a non-website approach:
start with **01**, then move to **03**.

If you are unsure whether to build a shared presence at all:
read **01** and decide from there.
